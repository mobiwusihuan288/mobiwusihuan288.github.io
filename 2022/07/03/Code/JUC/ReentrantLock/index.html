# ReentrantLock 介绍

  1. Java 中提供锁，一般就是 synchronized 和 lock 锁，ReentrantLock 是互斥锁，跟 synchronized 一样。

  2. ReentrantLock 使用

    注意点

    synchronized 在执行的过程中，会有 monitorenter 和 monitorexit 字节码指令，所以 synchronized 是自动释放锁的。

    ReentrantLock 执行 lock.lock 加锁，然后一定要在你的 try { } 里面去执行你的业务代码，因为你的业务代码可能会抛出异常。那么最终在 finally 中，记住一定把你的锁进行手动释放。

  3. 如何选择使用

    • 如果竞争比较激烈，推荐 lock 锁，效率更高

    • 如果几乎没有竞争，推荐 synchronized

      原因：

      synchronized 只有锁升级，当升级到重量级锁后，无法降级到轻量级、偏向锁。如果你的锁竞争特别激烈的话，那么只要用了 synchronize 那必然走重量级锁的套路，它肯定会把线程挂起。并且解锁的时候会涉及到线程的挂起，就存在用户态和内核态之间的转换，从而造成性能相对来说是比较低的。

      ReentrantLock 则不一样，它的内部是基于 CAS 以及 AQS 这种队列去排队，排不到的话，我才会把它通过 park 方法将线程挂起。即便挂起之后，当我的竞争没有那么激烈之后，依然是以 CAS 的方式在帮你竞争锁资源。

  4. 实现方式不同

    synchronized 基于对象实现，lock 锁基于 AQS+CAS 实现

  5. 是否公平角度考虑

    synchronized 是非公平锁，lock 是公平 + 非公平锁

  6. lock 锁提供的功能更完善

    • lock 可以使用 tryLock () 指定等待锁的时间，可以指定我等多久。如果等不到的话，我就直接返回一个 false 反之为 true
    • lock 锁还提供了 lockInterruptibly () 允许线程在获取锁的期间被中断。

lock 锁的使用相对 synchronized 成本更高。(学习成本)

# acquire 方法

我们发现不管是非公平还是公平的实现方式，都会调用 acquire (), 我们具体看一下它是干嘛的

acquire 是 AQS 提供一个业务方法，里面并没有实际的业务处理，都是在调用其他方法

  1. 调用 tryAcquire 方法：尝试获取锁资源（非公平、公平），拿到锁资源，返回 true，直接结束方法。 没有拿到锁资源，需要执行 && 后面的方法
  2. 当没有获取锁资源后，会先调用 addWaiter：会将没有获取到锁资源的线程封装为 Node 对象，并且插入到 AQS 的队列的末尾，并且作为 tail
  3. 继续调用 acquireQueued 方法，查看当前排队的 Node 是否在队列的前面，如果在前面（head 的 next），尝试获取锁资源，如果没在前面，尝试将线程挂起，阻塞起来！

# addWaiter 方法

在获取锁资源失败后，需要将当前线程封装为 Node 对象，并且插入到 AQS 队列的末尾

整个 addWaiter () 分为两个部分

  1. 第一次尝试把当前线程添加到 AQS 队列中
  1. 第一次添加失败，会走 enq (node); 方法以 for (;;) 的方式把当前线程添加到 AQS 队列中

t == null 高并发的情况下，可能执行 enq 的时候，如果 tail 为 null，我们就创建一个节点

无论是第一次添加成功，还是 enq 的方式添加成功整体流程可以分为三个部分，这三个步骤的 顺序 非常重要。

  1. node.prev = t; 将之前的 tail 节点设置为我的 pred

  1. compareAndSetTail(t, node) 以 CAS 的方式将 tail 指向当前线程

  1. t.next = node; 前节点 next 指向当前线程

# unparkSuccessor 方法

为什么唤醒线程时，为啥从尾部往前找，而不是从前往后找？

因为在 addWaiter 操作时，是先将当前 Node 的 prev 指针指向前面的节点，然后是将 tail 赋值给当前 Node，最后才是能上一个节点的 next 指针，指向当前 Node。

如果从前往后，通过 next 去找，可能会丢失某个节点，导致这个节点不会被唤醒～

如果从后往前找，肯定可以找到全部的节点。
